# `interpreter.cpp`

## Interpretation and Execution

`interpreter.cpp` acts as the execution engine for the application. It consumes the flattened sequential instructions generated by the AST parser, resolves the temporary placeholder tokens using a dynamic runtime map, and computes the final numeric solution.

---

## The Interpreter Logic

The file contains two main components: an entry-point router (`interpreter()`) and the primary execution loop (`evaluator()`).

### 1. Variable Storage and Tracking
Within `evaluator()`, state is managed via:
* **`variableStore`**: A `std::map<std::string, double>` that mimics physical CPU registers. It associates generated temporary placeholder tokens (e.g., `T1`, `T2`) with their calculated floating-point values.
* **`solution`**: A rolling `double` precision variable that continuously traps the latest sub-calculation. When the final instruction block finishes executing, this tracking variable holds the complete structural result of the expression.

### 2. The 5-Stride Token Decoding Loop
The evaluator iterates through the `orderedVector` by grouping the sequence in uniform strides of **5 tokens** (`n += 5`). Each stride maps directly to a discrete intermediate operation triplet:

1. `leftString` ($n$): The left operand literal or variable register.
2. `opSymbol` ($n+1$): The function name or operator symbol to invoke.
3. `rightString` ($n+2$): The right operand literal or variable register.
4. `arrow` ($n+3$): The structural delimiter token (`->`).
5. `targetValue` ($n+4$): The target placeholder token identifier (e.g., `T_n`).

### 3. Operand Processing & Stod Conversion
For each stride, the components are resolved into hard values:
* **Left & Right Value Resolution**: The interpreter checks if `leftString` and `rightString` exist as saved keys inside `variableStore`. If a key matches, its cached `double` value is pulled from the map. If it does not match, the string is treated as a literal number (e.g., `"4.3"`) and is converted to a runtime `double` using `std::stod`.

### 4. Mathematical Mapping
The `opSymbol` is cross-examined using structural branch checks to execute standard C++ standard library (`<cmath>`) or arithmetic primitives:

* **Binary Arithmetic**: Supports `*`, `/`, `+`, and `-` operations directly.
* **Exponents**: Maps the `^` symbol to `std::pow(leftNum, rightNum)`.
* **Unary Trigonometry**: Passes `rightNum` into hardware-level standard functions: `std::sin`, `std::cos`, or `std::tan`.
* **Inverse Trigonometry**: Evaluates reciprocal identities manually:
  * `sec` $\rightarrow \frac{1.0}{\cos(\text{rightNum})}$
  * `csc` $\rightarrow \frac{1.0}{\sin(\text{rightNum})}$
  * `cot` $\rightarrow \frac{1.0}{\tan(\text{rightNum})}$
* **Logarithms**: Maps `log` to common base-10 calculation (`std::log10`) and `ln` to natural base-$e$ calculations (`std::log`).

### 5. Assignment Update
Once `tempResult` is computed, it is recorded in the map at the index of `targetValue`:
$$\text{variableStore[targetValue]} = \text{tempResult}$$
This step feeds values forward so subsequent operations can parse and leverage previous computational states.

## Visual Execution Pipeline

```mermaid
graph TD
    Start([interpreter Called]) --> DebugIn[If debugMode: Print Input Token Vector]
    DebugIn --> CallEval[Invoke evaluator orderedVector]
    
    %% Loop Initialization
    CallEval --> LoopInit[Initialize variableStore Map<br>Set solution = 0.0]
    LoopInit --> LoopStart{n < orderedVector.size ?}
    
    %% Decoding Strides
    LoopStart -- Yes --> Decode[Extract Block Strides:<br>leftString = n<br>opSymbol = n+1<br>rightString = n+2<br>targetValue = n+4]
    
    %% Resolve Left
    Decode --> LeftCheck{Is leftString inside variableStore?}
    LeftCheck -- Yes --> AssignLeftVar[leftNum = variableStore leftString]
    LeftCheck -- No --> AssignLeftLit[leftNum = std::stod leftString]
    
    %% Resolve Right
    AssignLeftVar --> RightCheck{Is rightString inside variableStore?}
    AssignLeftLit --> RightCheck
    RightCheck -- Yes --> AssignRightVar[rightNum = variableStore rightString]
    RightCheck -- No --> AssignRightLit[rightNum = std::stod rightString]
    
    %% Compute Branches
    AssignRightVar --> OpMatch{Match opSymbol}
    AssignRightLit --> OpMatch
    
    OpMatch -->|exp: ^| PowCalc[tempResult = std::pow leftNum, rightNum]
    OpMatch -->|trig: sin/cos/tan| StandardTrig[tempResult = std::sin/cos/tan rightNum]
    OpMatch -->|inv trig: sec/csc/cot| InvTrig[tempResult = 1.0 / std::cos/sin/tan rightNum]
    OpMatch -->|logarithms: log/ln| LogCalc[tempResult = std::log10/log rightNum]
    OpMatch -->|basic: * / + -| BasicMath[tempResult = leftNum op rightNum]
    
    %% Store and Loop Shift
    PowCalc --> SaveState
    StandardTrig --> SaveState
    InvTrig --> SaveState
    LogCalc --> SaveState
    BasicMath --> SaveState
    
    SaveState[variableStore targetValue = tempResult<br>solution = tempResult] --> AdvanceStride[n += 5]
    AdvanceStride --> LoopStart
    
    %% Exit Paths
    LoopStart -- No --> ReturnSolution[Return solution]
    ReturnSolution --> DebugOut[If debugMode: Print Numeric Result]
    DebugOut --> End([Return Calculated Double Value])
